BUG-001

H2 Database v2.4.240 Throws Check Constraint Violation on Simple Inserts

Critical ✅ Fixed Backend P1
🧩 Component: Database / H2 🧪 Affected Tests: All Repository Tests (BE-002 → BE-060) 🔍 Discovered By: Automated Test Run (ProductRepositoryTest) 🌐 Environment: Spring Boot 4.0.3 / Java 17 / H2 in-memory
Root Cause
H2 version 2.4.240 introduced stricter check-constraint validation that conflicts with the Packora JPA schema. Simple INSERT statements (e.g., saving a Product or User) raised SQLState 23514 (check constraint violated), blocking all @DataJpaTest tests from running.
Steps to Reproduce
1. Upgrade H2 to 2.4.240 in pom.xml.
2. Run `./mvnw test` on any Repository test.
3. Observe: HibernateJdbcException → Check constraint "ORDERS_STATUS_CHECK" invalid.
✅ Fix Applied
Version-pinned H2 to 2.3.232 in backend/pom.xml:
2.3.232
📁 backend/pom.xml
📝 Notes
This is a known incompatibility. Spring Boot 4 auto-resolves to H2 2.4.x; explicit version override is required.
BUG-002

Frontend Tests Fail: TextEncoder / TextDecoder Not Defined in Jest Environment

Critical ✅ Fixed Frontend P1
🧩 Component: Test Environment / Jest 🧪 Affected Tests: All Component Tests (FE-022 → FE-048) 🔍 Discovered By: Automated Test Run (LoginPage.test.jsx) 🌐 Environment: React 19 / Jest 29 / jsdom / React Router v7
Root Cause
React Router v7 (used internally by React 19 test wrappers) requires the Web-standard TextEncoder and TextDecoder APIs. Jest's default jsdom environment does not include these, causing a ReferenceError at import time for any component that uses routing.
Steps to Reproduce
1. Create a test for LoginPage.test.jsx or any component wrapped in .
2. Run `npm test`.
3. Observe: ReferenceError: TextEncoder is not defined.
✅ Fix Applied
Added polyfills to src/setupTests.js:
const { TextEncoder, TextDecoder } = require('util');
Object.assign(global, { TextEncoder, TextDecoder });
📁 src/setupTests.js
📝 Notes
Polyfill must be placed before any React Router import in setupTests.js. Order matters.
BUG-003

Spring Boot 4 @DataJpaTest: application-test.properties Not Auto-Loaded

High ✅ Fixed Backend P2
🧩 Component: Test Configuration / Spring Boot 🧪 Affected Tests: All Repository Tests (BE-002 → BE-060) 🔍 Discovered By: Automated Test Run (UserRepositoryTest) 🌐 Environment: Spring Boot 4.0.3 / JUnit 5
Root Cause
Spring Boot 4 changed the auto-detection path for test properties. The file at src/test/resources/application.properties was not picked up for @DataJpaTest slice tests, causing tests to attempt a full datasource connection instead of using H2 in-memory.
Steps to Reproduce
1. Create a @DataJpaTest class without @ActiveProfiles annotation.
2. Run the test without application-test.properties in the correct location.
3. Observe: DataSource connection refused (attempts to connect to production DB).
✅ Fix Applied
Created src/test/resources/application-test.properties with:
spring.datasource.url=jdbc:h2:mem:testdb
spring.datasource.driver-class-name=org.h2.Driver
spring.jpa.hibernate.ddl-auto=create-drop
Added @ActiveProfiles("test") to all @DataJpaTest classes.
📁 backend/src/test/resources/application-test.properties
📝 Notes
All @DataJpaTest classes must declare @ActiveProfiles("test") explicitly in Spring Boot 4.
BUG-004

Single Table Inheritance (STI) Schema Missing Discriminator for SupportStaff Subtype

High ✅ Fixed Backend P2
🧩 Component: Repository / User / JPA 🧪 Affected Tests: BE-052, BE-053 (TicketRepositoryTest) 🔍 Discovered By: Automated Test Run (TicketRepositoryTest) 🌐 Environment: Spring Boot 4.0.3 / Hibernate 6 / JPA
Root Cause
The User entity uses Single Table Inheritance (@Inheritance(strategy = SINGLE_TABLE)). SupportStaff, a subtype required for Ticket association tests, was not included in the entity inheritance hierarchy, causing a MappingException when @DataJpaTest tried to build the schema.
Steps to Reproduce
1. Write TicketRepositoryTest with a SupportStaff user.
2. Run the test.
3. Observe: org.hibernate.MappingException: Unknown entity class.
✅ Fix Applied
Verified SupportStaff extends User with @DiscriminatorValue("SUPPORT_STAFF"). Added SupportStaff to the @DataJpaTest entity scan.
📁 backend/src/test/java/com/packora/backend/repository/TicketRepositoryTest.java
📝 Notes
STI subtypes must be on the classpath and visible to Hibernate for slice tests to work.
BUG-005

generate-test-report.js: Nested Template Literal Syntax Error Prevents HTML Generation

High ✅ Fixed Tooling P2
🧩 Component: Reporting Script 🧪 Affected Tests: All Tests (HTML Report Generation) 🔍 Discovered By: Manual script execution 🌐 Environment: Node.js / JavaScript
Root Cause
The HTML report block inside generate-test-report.js used unescaped backtick characters (`) within a template literal string. Node.js interpreted these as closing the outer template literal, causing a SyntaxError: Invalid or unexpected token at runtime.
Steps to Reproduce
1. Run `node generate-test-report.js`.
2. Observe: SyntaxError at the HTML content block.
✅ Fix Applied
Refactored HTML generation to use string concatenation via a for-loop (backendRowsHtml, frontendRowsHtml) instead of nested template literals. The outer template literal now contains only safe single-quoted and double-quoted HTML.
📁 generate-test-report.js
📝 Notes
Nested template literals in JS are valid but require careful escaping. Using string concatenation for large HTML blocks is safer.

BUG-008

API Token Retrieval: No Handling of localStorage SecurityError (Private Browsing)

Medium ⚠️ Open Frontend P3
🧩 Component: Utils / api 🧪 Affected Tests: FE-009 → FE-014 (api.test.js) 🔍 Discovered By: Code Review during api.test.js authoring 🌐 Environment: Browser (Private/Strict mode)
Root Cause
The getStoredToken() function in api.js catches malformed JSON but does not handle the case where localStorage is completely inaccessible (e.g., in private/incognito mode with strict settings, or when storage is disabled by browser policy). This throws an uncaught SecurityError.
Steps to Reproduce
1. Open app in a browser with localStorage disabled (about:config → dom.storage.enabled=false in Firefox).
2. Observe: Uncaught DOMException: SecurityError.
💡 Recommendation
Recommended: Wrap the localStorage.getItem() call in an additional try-catch for SecurityError.
📝 Notes
Low probability but worth hardening for production.
BUG-009

Order Repository: sumRevenue Query Returns Null Instead of 0 When No Orders Exist

Medium ⚠️ Open Backend P3
🧩 Component: Repository / Order 🧪 Affected Tests: BE-047 (OrderRepositoryTest — sumRevenue_shouldExcludeCancelled) 🔍 Discovered By: Test analysis during OrderRepositoryTest authoring 🌐 Environment: Spring Boot 4 / JPA
Root Cause
The @Query for summing revenue uses SUM() which returns NULL when the result set is empty (SQL standard behavior). The calling service layer does not null-check this, which can lead to a NullPointerException when there are no non-cancelled orders.
Steps to Reproduce
1. Clear all orders from the DB.
2. Call the revenue sum endpoint.
3. Observe: NullPointerException in service layer (or null returned to API consumer).
💡 Recommendation
Recommended: Use COALESCE(SUM(o.totalAmount), 0) in the JPQL query, or null-check the return value in the service layer.
📝 Notes
Test currently passes because test always seeds at least one valid order before asserting.
BUG-010

PaymentSuccess: Transaction ID Exposed in Plain URL Parameter (No Encoding)

Low ⚠️ Open Frontend P4
🧩 Component: PaymentSuccess 🧪 Affected Tests: FE-045, FE-046 (PaymentSuccess.test.jsx) 🔍 Discovered By: Code Review during PaymentSuccess.test.jsx authoring 🌐 Environment: React 19 / Browser
Root Cause
The PaymentSuccess component reads the transaction ID directly from the URL search parameter (?txn=...) and renders it unencoded. If the transaction ID contains special characters, it can break URL parsing. Additionally, raw transaction IDs in URLs are visible in browser history and server logs.
Steps to Reproduce
1. Complete a payment with a transaction ID containing & or # characters.
2. Observe: URL parameter is truncated/misread.
💡 Recommendation
Recommended: Encode the txn parameter with encodeURIComponent() when building the URL, and decode with decodeURIComponent() on read.
📝 Notes
Security-adjacent but low severity in practice as transaction IDs are typically alphanumeric.
